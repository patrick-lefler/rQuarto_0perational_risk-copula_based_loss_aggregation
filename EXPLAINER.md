# Understanding "The Diversification Question"

Imagine a company keeps a rainy-day fund big enough to cover its worst-case losses in a bad year, just in case. The question behind this project is simple to ask and surprisingly hard to answer: how big does that fund actually need to be?

NexaCore Financial Technologies — a made-up fintech company used to make this analysis concrete — tracks five different kinds of problems that can cost it money each year, things like internal fraud, computer system outages, and customer complaints. The easy way to size a rainy-day fund is to figure out the worst-case cost for *each* problem separately, then add all five worst cases together. But that adds up to a huge number, because it assumes all five things go wrong at their worst, in the same year, at the same time. In real life, that almost never happens.

This project tests a smarter way to size the fund: instead of assuming everything goes wrong at once, it asks how likely the different problems actually are to happen *together*. Most of the five problem types here have nothing to do with each other — a computer outage doesn't make a customer complaint more likely. But one pair does: a serious system outage tends to cause extra processing mistakes in the same year, since IT problems create knock-on effects elsewhere in the company.

To study this, ten years of made-up (synthetic) loss data were generated for each problem type, similar to rolling weighted dice many times and keeping track of the results. A mathematical tool called a "copula" was then used to measure exactly how strongly the outage and processing-mistake categories actually move together, based on that ten-year history. A computer then ran the whole scenario 50,000 times to see what a realistic worst-case year looks like when problems are allowed to move together the way the data suggests, instead of all peaking at once.

The result: the smarter rainy-day fund came out about 30% smaller than the naive add-everything-up approach, while still covering the worst 1-in-1,000-year scenario. That's a meaningful amount of money that doesn't need to sit around unused.

The most interesting finding, though, showed up when several different versions of the "copula" math tool were tried instead of just one. The size of the savings barely changed no matter which version was used. That's surprising at first, since you'd expect the answer to depend heavily on getting the math exactly right. The reason it doesn't: only one pair of the five problem types is actually connected to each other. The other three are unrelated by design, and most of the savings simply comes from the common-sense fact that unrelated problems rarely all go wrong in the same year. Picking the "best" math model for the one connected pair turned out to matter far less than knowing *which* pairs are connected in the first place — a lesson that applies well beyond this one example.
